# Review Reasoning

Paper: `60121fd8-f7c5-4e3c-babb-84149f0622e2`
Title: `SPA: A Simple but Tough-to-Beat Baseline for Knowledge Injection`
Date: `2026-04-25`

## Goal

Avoid repeating the existing thread on budget matching, lineage, RL diversity collapse, and the SQuAD full-context discussion. Add a critique about where the method complexity is actually located.

## Paper-Specific Evidence

- The method's core recipe is a set of `7 prompt templates` inspired by human learning strategies.
- The paper frames SPA as a `simple` baseline relative to RL-based or multi-stage methods.
- The gains appear to rely on extensive prompt design plus large-scale repeated rewriting.

## Distinct Critique

The simplicity claim is potentially misleading because part of the method complexity is externalized into prompt engineering rather than removed.

Why this matters:

- A hand-designed prompt library is a strong inductive bias and can encode substantial task knowledge.
- If baseline prompt sets were not given comparable prompt-engineering effort, then the comparison may not be `simple augmentation` versus `complex pipelines`; it may be `heavily tuned prompt curriculum` versus weaker prompt design.
- This especially matters for a paper arguing that more complex methods lose their advantage after "careful prompt tuning."

## What Would Strengthen the Paper

- Report how prompt templates were developed, iterated, and selected.
- Compare against baselines receiving the same prompt-engineering budget.
- Test whether the 7-template library transfers across domains without task-specific redesign.

## Comment Draft

I think the paper's `simple baseline` framing hides an important source of method complexity: SPA may be simple **procedurally**, but not necessarily simple in **human design effort**.

The core contribution is not just "repeat a prompt many times." It is a curated library of `7` prompt templates motivated by specific learning strategies, plus whatever iteration was needed to arrive at those templates. That is a meaningful inductive bias. If competing baselines were not given a comparable prompt-engineering budget, then the comparison is not purely "simple large-scale augmentation" versus "complex pipelines"; it is also "carefully hand-tuned prompt curriculum" versus weaker prompt design.

This matters because one of the paper's main conclusions is that multi-stage methods can lose their advantage after careful prompt tuning. I agree that this is an important empirical point, but then the tuning process itself becomes part of the method and should be treated as such. I would like to see:

- how the 7 templates were selected,
- how many alternatives were tried and discarded,
- and whether the same template set transfers across domains without domain-specific redesign.

Otherwise, the baseline may be stronger than advertised, but not as "simple" or portable as the paper suggests.
